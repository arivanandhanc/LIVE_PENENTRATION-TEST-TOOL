What it is

Arivanandhan SecTools is an active web-application penetration-testing suite. Unlike a passive header checker, it first crawls the target to map its real attack surface, then runs a battery of 25 modules — including active injection tests that send safe, uniquely-marked payloads and confirm vulnerabilities from the application's actual response.

It is delivered as a single Node.js process: an Express backend exposing a small JSON API and a zero-build vanilla-JavaScript frontend. Scans stream their progress live to the browser over Server-Sent Events (SSE).

Defensive by design. Every scan requires an explicit authorisation confirmation. Active tests are non-destructive and marker-based. The hosted service refuses to scan loopback/private hosts to prevent SSRF abuse.

How the testing works

A scan runs as a pipeline. Each stage feeds the next:

Resolve & guard

The target URL is parsed and DNS-resolved. On the hosted service, any target resolving to a loopback/private/link-local address is rejected (SSRF protection).

Crawl & map the surface

The crawler performs a same-origin breadth-first spider (bounded by the profile's page/depth budget), parses forms and query parameters, seeds extra paths from robots.txt and sitemap.xml, and mines JavaScript files for hidden API endpoints. The output is a surface — the set of pages, forms, and injection points.

Run the modules

The orchestrator runs each selected module in turn, passing it a shared context (the target, the crawl surface, request budgets, and a logger). Passive modules inspect responses; active modules send safe, uniquely-marked payloads into each discovered injection point and inspect the result for confirmation.

Confirm, score & report

Each confirmed issue becomes a normalised finding with severity, CVSS, CWE, OWASP mapping, evidence, and remediation. Findings are tallied into a risk score and rendered into the report.

Why detection is reliable (low false-positives)

  • Unique markers. XSS uses a random breakout token; SSTI uses a random arithmetic product (e.g. 73×74) so a literal echo can't be mistaken for evaluation.
  • Reflection guards. SSRF only fires when the response contains fetched content that is not just our echoed payload URL.
  • Differential analysis. Boolean SQLi compares TRUE vs FALSE response sizes against a baseline.
  • Soft-404 baselines. Content/file discovery compares against a known-missing page so catch-all apps don't produce phantom findings.

Authenticated scanning

Supply a session and the crawler plus every active module will test behind your login. You can provide any combination of a Cookie header, a Bearer token, and arbitrary extra headers.

Internally this uses Node's AsyncLocalStorage: the whole scan runs inside an auth context, so every HTTP request automatically carries the credentials — and concurrent scans never leak auth to each other.

Reports

Reports open with an executive summary carrying a 0–100 risk score and A–F grade, then a confidentiality notice, document control, risk-rating methodology, reconnaissance & attack surface, a findings table, the OWASP Top 10 (2021) coverage matrix, a remediation roadmap, and detailed per-finding write-ups with CVSS, CWE, evidence and references. Export to PDF (one-click print-to-PDF — no headless browser required) or JSON.

Self-hosting

Runs as a single Node process — ideal for a small VPS (not serverless, because port recon and the burst check need raw TCP sockets and long-lived SSE).

git clone https://github.com/arivanandhanc/LIVE_PENENTRATION-TEST-TOOL.git cd LIVE_PENENTRATION-TEST-TOOL npm ci --omit=dev cp .env.example .env # keep ALLOW_PRIVATE=false on a public host node server.js # http://localhost:3000

The repository ships a Dockerfile, a systemd unit, and an nginx config (with SSE-safe proxy settings) under deploy/.
